## Feature Under Test

When the device loses connectivity to the Memfault platform, `memfault_log_trigger_collection()`
is called immediately to freeze the Memfault RAM log buffer at the moment of loss.
This preserves the disconnect-context log snapshot so it is uploaded intact on the
next successful connection, enabling post-mortem debugging of the connectivity failure itself.

Three independent failure layers are covered:

| Layer | Trigger source | Code path |
|---|---|---|
| L1 — Wi-Fi deassociation | `WIFI_STA_DISCONNECTED` on WIFI_CHAN | `memfault_wifi_listener` |
| L2 — IP-layer loss | `NETWORK_NOT_READY` on NETWORK_CHAN | `memfault_network_listener` |
| L3 — Internet down (IP/Wi-Fi up) | Upload failure in `on_connect()` / periodic timer | `memfault_zephyr_port_post_data()` return value |

## TC-L1 — Wi-Fi Deassociation

**Goal:** confirm `WIFI_STA_DISCONNECTED` freezes the log buffer and the snapshot uploads on reconnect.

**Distinguishing characteristic:** both `WiFi DISCONNECTED` and `Network connectivity lost` logs appear — the NETWORK_CHAN fires as a side effect of the Wi-Fi disconnect.

### Pre-condition

Device is Wi-Fi connected and has been logging for ≥ 30 seconds (to populate the log buffer with meaningful content around the disconnect event).

### Procedure

| Step | Action | Expected UART output |
|---|---|---|
| 1 | Confirm device connected | `[WiFi] WiFi is connected!` visible in recent log |
| 2 | Deauth device from router (see command below) | — |
| 3 | Observe disconnect on UART | `=== WiFi DISCONNECTED (reason: 3)` |
| 4 | Observe freeze | `memfault_core: Network connectivity lost — freezing Memfault logs` |
| 5 | Confirm **no** upload attempt during offline period | No `Sending already captured data` line while disconnected |
| 6 | Let device auto-reconnect | `[WiFi] WiFi is connected!` |
| 7 | Observe upload on reconnect | `memfault_core: Sending already captured data to Memfault` |

**Router command to deauth:**

```python
import paramiko, os
ssh = paramiko.SSHClient()
ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
ssh.connect("192.168.1.1", username="admin", password=os.getenv("ROUTER_PASS"))
DEVICE_MAC = "xx:xx:xx:xx:xx:xx"   # replace with actual
ssh.exec_command(f"wl -i eth6 deauthenticate {DEVICE_MAC}")
```

Alternatively: power-cycle the AP or run `service restart_wireless` on the router.

### Pass Criteria

| # | Criterion |
|---|---|
| P1 | `Network connectivity lost — freezing Memfault logs` appears on UART **before** any reconnect |
| P2 | No `Sending already captured data` log appears while offline |
| P3 | After reconnect: `Sending already captured data to Memfault` appears |
| P4 | Memfault dashboard → Issues/Logs shows the log snapshot ending at the disconnect moment |
| P5 | Memfault dashboard → Metrics → `connectivity_connected_time_ms` shows a gap matching the offline window |

### Fail Indicators

- `Network connectivity lost` does **not** appear → WIFI_CHAN listener not firing
- Logs uploaded during offline period → freeze not working, buffer still rolling
- Dashboard logs are empty after reconnect → `memfault_log_trigger_collection()` called too late

## TC-L2 — IP-Layer Loss (DHCP Expiry / Address Removal)

**Goal:** confirm `NETWORK_NOT_READY` alone (without `WIFI_STA_DISCONNECTED`) freezes the log buffer.

**Distinguishing characteristic:** `WiFi DISCONNECTED` does **not** appear in UART; only `Network connectivity lost` appears. Wi-Fi remains associated throughout.

### Pre-condition

Device is connected. Note the current DHCP lease expiry time if possible (usually 24 h; shorten if needed via router config).

### Procedure

| Step | Action | Expected UART output |
|---|---|---|
| 1 | Block DHCP from device and evict lease (see script below) | — |
| 2 | Wait for DHCP renewal attempt (~lease-time/2 by default, or immediately after eviction) | `[DHCP] DHCP client started` |
| 3 | Renewal blocked → IP removed by Zephyr | `[IPv4] IP address removed from interface` |
| 4 | Observe freeze — NETWORK_CHAN fires | `memfault_core: Network connectivity lost — freezing Memfault logs` |
| 5 | Confirm **no** `WiFi DISCONNECTED` in log during this sequence | *(absence of line)* |
| 6 | Restore DHCP (see script below) | `[DHCP] DHCP bound` |
| 7 | Observe upload | `memfault_core: Sending already captured data to Memfault` |

**Router script:**

```python
import paramiko, os, time
ssh = paramiko.SSHClient()
ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
ssh.connect("192.168.1.1", username="admin", password=os.getenv("ROUTER_PASS"))
DEVICE_MAC = "xx:xx:xx:xx:xx:xx"   # replace

# Block DHCP packets from device
ssh.exec_command(f"iptables -I FORWARD -m mac --mac-source {DEVICE_MAC} -p udp --dport 67 -j DROP")
ssh.exec_command(f"iptables -I INPUT  -m mac --mac-source {DEVICE_MAC} -p udp --dport 67 -j DROP")
# Evict the lease to force immediate renewal attempt
ssh.exec_command("kill -HUP $(pidof dnsmasq)")

print("Waiting 90 s for lease expiry and IP removal on device side...")
time.sleep(90)

# Restore DHCP
ssh.exec_command(f"iptables -D FORWARD -m mac --mac-source {DEVICE_MAC} -p udp --dport 67 -j DROP")
ssh.exec_command(f"iptables -D INPUT  -m mac --mac-source {DEVICE_MAC} -p udp --dport 67 -j DROP")
print("DHCP restored — device should obtain a new IP.")
```

### Pass Criteria

| # | Criterion |
|---|---|
| P1 | `Network connectivity lost — freezing Memfault logs` appears on UART |
| P2 | `WiFi DISCONNECTED` does **not** appear in the UART log during steps 1–5 |
| P3 | After DHCP restore: `Sending already captured data to Memfault` appears |
| P4 | Dashboard logs show the snapshot taken at the moment of IP loss |

### Fail Indicators

- `WiFi DISCONNECTED` appears → IP loss caused a full Wi-Fi disconnect (test not isolating L2; try a shorter block window)
- `Network connectivity lost` does **not** appear → NETWORK_CHAN listener not wired, or `NET_EVENT_IPV4_ADDR_DEL` not reaching Zephyr

## TC-L3 — Internet Down, IP and Wi-Fi Still Up

**Goal:** confirm that when Memfault servers are unreachable but Wi-Fi and IP are intact, the upload failure is detected and logged, and the log buffer is frozen by the periodic upload timer's call to `memfault_log_trigger_collection()`.

**Distinguishing characteristic:** no `WiFi DISCONNECTED` and no `Network connectivity lost` on UART. The failure signal is `Memfault upload failed: <errno>`.

**Timing note:** the freeze happens inside the periodic upload timer (every 60 s), not immediately. There is a window of up to 60 s where the buffer is still rolling after internet is cut.

### Procedure

| Step | Action | Expected UART output |
|---|---|---|
| 1 | Block all outbound traffic from device (see script below) | — |
| 2 | Wait for next periodic upload attempt (≤ 60 s) | `memfault_core: Sending already captured data to Memfault` |
| 3 | Upload fails | `memfault_core: Memfault upload failed: -1` (or similar errno) |
| 4 | Note: log buffer is now frozen by periodic timer's `memfault_log_trigger_collection()` | *(no explicit freeze log for this path)* |
| 5 | Continue blocking for 2–3 minutes | Periodic timer fires again: `Memfault upload failed: -1` each cycle |
| 6 | Restore internet access (see script below) | — |
| 7 | Next periodic timer succeeds | `memfault_core: Sending already captured data to Memfault` (no failure log) |
| 8 | Dashboard updated | log snapshot and `sync_memfault_failure` metric visible |

**Router script:**

```python
import paramiko, os, time
ssh = paramiko.SSHClient()
ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
ssh.connect("192.168.1.1", username="admin", password=os.getenv("ROUTER_PASS"))
DEVICE_MAC = "xx:xx:xx:xx:xx:xx"   # replace

# Block all outbound traffic from device (simulates full internet outage)
ssh.exec_command(f"iptables -I FORWARD -m mac --mac-source {DEVICE_MAC} -j DROP")
print("Internet blocked. Waiting 75 s for periodic upload attempt...")
time.sleep(75)

# Restore internet
ssh.exec_command(f"iptables -D FORWARD -m mac --mac-source {DEVICE_MAC} -j DROP")
print("Internet restored.")
```

### Pass Criteria

| # | Criterion |
|---|---|
| P1 | `Memfault upload failed:` log appears on UART after blocking internet |
| P2 | `WiFi DISCONNECTED` and `Network connectivity lost` do **not** appear |
| P3 | After restore: next periodic timer attempt succeeds (no failure log) |
| P4 | Memfault dashboard → Metrics → `sync_memfault_failure` incremented by the number of failed periodic cycles |
| P5 | Dashboard logs show a snapshot from around the time internet was blocked |

### Fail Indicators

- No `Memfault upload failed` log → `memfault_zephyr_port_post_data()` return value still not checked (code regression)
- `sync_memfault_failure` metric not present in dashboard → `CONFIG_MEMFAULT_METRICS_MEMFAULT_SYNC_SUCCESS` not enabled

## TC-L4 — WiFi AP Not Available (Credentials Stored in NVS)

**Goal:** confirm the device survives the full WiFi timeout + reconnect retry loop without a stack overflow or reboot when the provisioned AP is unreachable.

**Background:** the reconnect path (`wpa_cli_cmd_remove_network` on `sysworkq`) allocates ~4488 bytes of stack depth on nRF54LM20DK. With `CONFIG_SYSTEM_WORKQUEUE_STACK_SIZE` set too small, a USAGE FAULT (RESETREAS=0x40) occurs silently during the first reconnect attempt — the fault dump is hidden by `LOG_MODE_DEFERRED`. This test is the primary validation that the stack sizing fix holds across boards.

### Pre-condition

Device is provisioned with valid WiFi credentials (stored in NVS) but the AP is **not available** (powered off, SSID renamed, or device physically out of range). Do **not** erase NVS — the credentials must be present.

### Procedure

| Step | Action | Expected UART output |
|---|---|---|
| 1 | With AP powered off, flash firmware or reboot device | `[WiFi] WiFi credentials detected, attempting to connect` |
| 2 | Wait for connection timeout (~30 s) | `[WiFi] Reason: Connection timed out (-ETIMEDOUT)` |
| 3 | Observe retry scheduling | Any of: `scheduling retry` / `attempting to connect` / `WiFi credentials detected` |
| 4 | Confirm **no crash or reboot** | No `***** USAGE FAULT *****`, no `RESETREAS`, no `--- N messages dropped ---` followed by reboot |
| 5 | Wait for 2 full retry cycles (~2 min total) | Two `Connection timed out` lines, each followed by a retry attempt |
| 6 | Power on the AP | — |
| 7 | Device auto-connects | `[WiFi] WiFi is connected!` |
| 8 | Normal operation resumes | `[Memfault] Sending already captured data` or periodic log output |

**Confirm no reboot occurred** (step 4 — run immediately after the first timeout):
```python
# Read RESETREAS via Zephyr shell to verify no fault reset
uart:~$ nrf_regs resetreas   # should show 0x0 (clean)
```

Or from UART log: if `RESETREAS: 0x00000040` appears, the stack overflow fix is not in effect.

### Pass Criteria

| # | Criterion |
|---|---|
| P1 | `Connection timed out (-ETIMEDOUT)` appears — device attempted the connect |
| P2 | **No** `***** USAGE FAULT *****` in UART log at any point |
| P3 | **No** `--- N messages dropped ---` followed by silent reboot |
| P4 | Device continues running through ≥ 2 full retry cycles |
| P5 | Device connects successfully once AP is restored |

### Fail Indicators

- `***** USAGE FAULT *****` or silent reboot after timeout → `CONFIG_SYSTEM_WORKQUEUE_STACK_SIZE` too small (see Mode C4–C5 in `chsh-sk-ncs-debug` for diagnosis)
- Device reboots silently (only `--- N messages dropped ---` visible) → enable `CONFIG_LOG_MODE_IMMEDIATE=y` to expose the hidden fault dump, then diagnose with addr2line
